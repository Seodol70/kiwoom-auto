# /analyze-trades — 거래 결과 분석 스킬

## 목적

매일 장 마감 후 자동매매 결과를 정량적으로 분석하여:

1. 어떤 신호 유형이 수익을 냈는지
2. 어떤 필터가 가장 많이 신호를 차단했는지
3. 조정이 필요한 파라미터는 무엇인지

를 데이터 기반으로 도출한다.

## 권장 실행 모델

이 스킬은 **Haiku** 또는 **Sonnet**으로 실행해도 충분하다. 정해진 워크플로우(로그 grep → 정규식 매칭 → 통계 집계 → 메모리 저장)이므로 Opus는 과잉이다. 실제 실행 시 사용자가 모델을 신경 쓰지 않아도 되도록 워크플로우를 단순하게 유지한다.

## 입력 인자

- **없음**: 가장 최근 영업일 자동 추론 (`logs/position.log` 마지막 라인의 날짜 사용)
- **YYYY-MM-DD**: 해당 날짜 거래만 분석

## 워크플로우

### Step 1: 분석 대상 날짜 결정

```python
# 인자 없으면 logs/position.log 마지막 거래일 추출
# tail -1 logs/position.log → 날짜 파싱
```

다음 명령으로 날짜 확인:
```bash
tail -5 logs/position.log
```

가장 최근 `[포지션생성]` 또는 `[포지션청산]` 라인의 날짜를 분석 대상으로 사용.

### Step 2: 데이터 수집 (로그 4종 grep)

분석 대상 날짜를 `$DATE`라 할 때:

```bash
# 1. 포지션 생성/청산 (실제 매매)
grep "$DATE" logs/position.log

# 2. 신호 수신 + 주문 (신호→매수 흐름)
grep "$DATE" logs/order.log | grep -E "신호수신|매수|매도"

# 3. 진입 거절 (필터 차단 사유)
grep "$DATE" logs/kiwoom_auto.log | grep -E "진입거절|RS필터|AI필터|시장.*급락|냉각기"

# 4. 신호 발생 (스캐너 출력)
grep "$DATE" logs/scanner.log 2>/dev/null | head -200
```

### Step 3: 통계 집계 (구조화)

다음 4개 섹션으로 집계:

#### A. 거래 결과 (실현 손익)
- 신호별 (BREAKOUT / PULLBACK / JDM_GC_OVERRIDE / JDM_ENTRY) 진입/청산 수
- 손익률 평균 / 최대 / 최소
- 승률 (수익 청산 / 전체 청산)
- 시간대별 분포 (오전/점심/오후)

#### B. 진입 거절 사유 (필터 차단 패턴)
- `should_entry` 거절 (포지션 풀, 가용 예수금 부족, 시간 외 등)
- `AI필터` 거절 (예상승률 < 임계값)
- `RS필터` 거절 (RS < 임계값)
- `시장급락` 차단 (KOSPI/KOSDAQ -3.0% 이상)
- `냉각기` 차단 (연속 손절 후)
- 각 사유별 차단 횟수 + 종목 샘플 3개

#### C. 활성 신호 vs 진입 신호
- 발생한 신호 총 개수
- 실제 매수까지 진행된 비율 (퍼널)
- 손실 발생 단계 식별

#### D. 시장 조건
- 그날 KOSPI/KOSDAQ 등락
- 시장 급락 임계값 도달 시점
- 냉각기 발동 시점

### Step 4: 터미널 출력

다음 형식으로 사용자에게 표시:

```
========================================
📊 거래 분석 — 2026-05-12 (오늘)
========================================

【거래 결과】
  • 진입: 4건 / 청산: 4건
  • 승률: 25% (1승 3패)
  • 평균 손익: -0.34%
  • 최대 수익: KBI메탈 +0.61%
  • 최대 손실: 노브랜드 -1.95%

  신호별:
  - BREAKOUT  진입 2 / 승률 50% / 평균 -0.67%
  - PULLBACK  진입 2 / 승률 0%  / 평균 -0.10%

【필터 차단 통계】
  • should_entry: 12건 차단
    - 포지션 풀: 8건
    - 가용 예수금 부족: 4건
  • RS필터: 0건 (모든 신호 통과)
  • AI필터: 비활성
  • 시장급락: 6건 차단 (-3.0% 초과)
  • 냉각기: 3건 차단 (12:42~12:57)

【신호 퍼널】
  발생: 47개 → 신호: 18개 → 진입: 4건 (8.5%)

【조정 제안】
  → 시장급락 임계값 -3.0% → -3.5% 검토
    이유: 6건 차단 중 4건이 -3.0~-3.3% 범위 (강한 종목)
  → 냉각기 시간 15분 → 10분 검토 (5회→3회 진입 기회)

저장 위치: memory/phase9_2026-05-12_analysis.md
========================================
```

### Step 6: 필터 조정 제안 (있을 경우 사용자 확인)

조정 제안이 있으면 다음 형식으로 확인:

```

1. scanner/config.py — market_crash_limit: -3.0 → -3.5
   근거: 6건 차단 중 4건이 -3.0~-3.3% 범위였고,
         그 중 2건은 RS필터까지 통과한 강세 종목

2. scanner/config.py — cooldown_duration_minutes: 15 → 10
   근거: 12:42~12:57 냉각기 중 3개 유효 신호 누락

적용할까요?
- [1] 모두 적용
- [2] 1번만 적용
- [3] 2번만 적용
- [4] 적용 안 함 (기록만)
```

사용자 응답에 따라:
- 적용 시: `Edit` 도구로 `scanner/config.py` 수정 + 변경 이력 메모리 저장
- 적용 안 함: 메모리에 "**제안됨 (미적용)**" 표시로 기록

## 주의사항

1. **로그가 없는 경우**: `logs/position.log`에 해당 날짜 거래가 없으면 "분석 대상 거래 없음" 안내 후 종료.

2. **로그 크기 제한**: 로그가 매우 크면 `grep` 결과를 `head -500` 정도로 제한. 그 이상은 핵심 패턴만 추출.

3. **메모리 중복 방지**: 같은 날짜 분석을 두 번 실행하면 기존 파일 덮어쓰기 (Write 도구 사용).

4. **사용자 의사결정 권한**: 필터 조정은 반드시 사용자 확인 후에만 적용. 자동 적용 금지.

5. **외부 시스템 변경 없음**: 분석은 로컬 로그만 사용. 키움 API 호출 금지.
